现在很多 APP 有给 AI 打电话的能力，但是只有人单方面打断 AI，AI 并不能打断人，其实体感上还是和真人打电话有差异。

我想了下在 Agent 工程侧做一些工作好像就可以实现这个能力：

1. 开始打电话时，和主 Agent 交互
2. 如果用户选择“打断模式”或者“真实模式”，则开启打断模式
3. 打断模式下，用户在说话过程中会通过**节流**的形式发送已经说的内容给新的 Sub Agent
4. Sub Agent 收到用户消息后，推理，如果觉得用户需要打断，则调用打断工具给出理由
5. 客户端收到打断信号后，停止用户的语音输入，强制播放 AI 返回的理由，除非用户再次打断

举个例子好理解：

**用户**：我觉得A不是B，因为A的长宽高和B完全不一致，xxx...

然后节流每隔x秒或者每隔x字发送一次给 Sub Agent

**发送给 Sub Agent 的消息 1**：我觉得

**发送给 Sub Agent 的消息 2**：我觉得A不是B，因为

**发送给 Sub Agent 的消息 3**：我觉得A不是B，因为A的长宽高

**发送给 Sub Agent 的消息 4**：我觉得A不是B，因为A的长宽高和B完全不一致

**Sub Agent**：

```json
{
  "tool_name": "interrupt",
  "tool_args": {
    "reason": "不，你说错了，A和B的长宽高是一样的，因为xxx"
  }
}
```

此时客户端收到打断工具调用后，真正打断了用户的语音输入，并开始播放 Sub Agent 返回的理由，后续用户也可以打断这个。

**注意**：这里的用户再次打断/不打断发送给主 Agent 时，需要用户之前被打断的输入和 Sub Agent 的打断时的理由也一起发送给主 Agent 作为 Context。

同时，主 Agent 虽然不会参与打断逻辑，但是要通过 System Prompt 来让他产生是他打断的感觉，类似：

> **你具有打断能力，当你觉得用户需要打断时，你会打断用户，所以在历史中会看到你打断用户的话语**

这样主 Agent 看到历史中的打断记录，就会觉得是自己打断的，继续走正常的逻辑。

同时，Sub Agent 的 System Prompt 也需要有之前的 history，才能精准地推理要不要打断。

不过这里只用一个 Agent 应该也是可以的，就是上下文管理要写好逻辑。
