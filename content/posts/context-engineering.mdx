# 一、上下文工程

## 1.1 什么是上下文工程

从 Prompt Engineering（提示词工程）到 Context Engineering（上下文工程）的范式转变，是 LLM 应用/Agent 发展过程中的一个重要趋势。

Agent 的极简定义可以由如下等式表示：

![Agent 定义](/posts/context-engineering/agent-def.png)

将 Agent 类比为一个人，LLM 是他的大脑或者意识/智慧，Tools 代表手、脚以及其他一切外部工具，如此便具备了【完成任务】这一能力。但是，人虽有智慧，也需要特定的知识、对周围环境的判断以及了解身边的工具的用途才能更好地、更准确地完成特定的任务，这些知识、环境信息和工具说明以及这里未提及的一些其他信息共同组成了需要灌输给这个人的上下文。

Context Engineering 即是指通过**精心构造和管理模型所看到的全部上下文信息**，来影响和优化模型的行为与输出。上下文主要包括如下内容：

![上下文组成部分](/posts/context-engineering/context-parts.png)

## 1.2 Prompt Engineering 与 Context Engineering 的关系

**Prompt Engineering 是 Context Engineering 的一个子集**。可以说，prompt 是上下文中的一部分，通常是 System Prompt 与 User Message。

**Context Engineering 是更全面、更系统化的视角**，它关注的是模型所接收到的**全部环境信息**如何影响输出，强调对信息的组织、管理和优化，由 Prompt Engineering 范式向 Context Engineering 范式转变是工程化程度提高的体现。

| 维度     | Prompt Engineering     | Context Engineering    |
| -------- | ---------------------- | ---------------------- |
| 关注点   | 单个提示词的设计       | 整体输入上下文的管理   |
| 控制方式 | 显式指令引导           | 通过信息结构与语境引导 |
| 适用阶段 | 早期探索、简单任务     | 复杂应用、生产环境     |
| 知识利用 | 依赖模型已有知识       | 可注入实时/专业知识    |
| 稳定性   | 对 prompt 敏感，易波动 | 更稳定，可系统化设计   |
| 未来趋势 | 仍是基础技能           | 构建可靠 AI 应用的关键 |

# 二、业界常见技巧

### 2.1 Offload 卸载

**一句话总结**：利用文件系统/sandbox 作为天然载体，将 Context 中的部分内容卸载至其中，并在 LLM 觉得需要时可以检索并找回相关的内容（可被恢复的压缩）

参考：https://www.anthropic.com/engineering/multi-agent-research-system

> As conversations extend, standard context windows become insufficient, necessitating intelligent compression and memory mechanisms. We implemented patterns where agents summarize completed work phases and store essential information in external memory before proceeding to new tasks. When context limits approach, agents can spawn fresh subagents with clean contexts while maintaining continuity through careful handoffs. Further, they can retrieve stored context like the research plan from their memory rather than losing previous work when reaching the context limit. This distributed approach prevents context overflow while preserving conversation coherence across extended interactions.
>
> 我们实施了这样的模式：智能体会总结已完成的工作阶段，并在开始新任务之前将关键信息存储到外部存储器中。当接近上下文限制时，智能体可以生成具有干净上下文的新子智能体，同时通过精心的交接来保持连贯性。此外，它们可以从自己的存储器中检索已存储的上下文（如研究计划），而不会在达到上下文限制时丢失之前的工作。这种分布式方法可以防止上下文溢出，同时在长时间交互中保持对话的连贯性。
>
>                                                                                                                                                                              ——— How we built our multi-agent research system, Anthropic

### 2.1.1 Memory

将 Long-term Memory/userMemory/模型生成的 TODO List/读取的大型文件的内容（如一个 pdf）等存储在文件系统，在 LLM 需要时去搜索读取文件以快速获取相关信息。

例如，在项目目录执行 Claude Code 的 init 命令时生成的 `CLAUDE.md` 文件：

![CLAUDE.md](/posts/context-engineering/claude-md.png)

Claude Code 在执行具体任务时，可能会首先搜索/查看这个文件的相关信息，找到一些上下文，然后再执行对应的任务。

### 2.1.2 Tools

在 Context 中，描述清晰、参数详尽的 tools declaration 往往占据很大一部分。尤其是在 MCP 协议推出后，膨胀了这一现象。因此，业界提出了 **progressive disclosure（渐进式披露）** 这样的概念。

如上所述，一个好的 declaration 是一个非常复杂的 JSON Schema，尤其是当工具的参数非常多时，JSON 数据会膨胀非常大，这会对模型造成迷惑。各大 Agent 平台对此问题一般有以下两点思路：

1. 尽量使用元工具。如 runInTerminal，参数只有一个，即 command，但是可以囊括许多事情。这里的一个点是 LLM 天然具备很强的代码能力，其直接写代码完成某个任务（先写个脚本然后在终端里运行再拿到输出）的完成度甚至超过理解开发者为其精心设计的复杂工具并调用

2. 复杂的能力需要渐进式的披露。如 Claude Code 的 skills，初始情况下只发送给 LLM 极少 skill 的元信息，如果 LLM 决定他要使用这个技能，再去读取被 offload 在文件系统中的 SKILL.md 等文件，从而避免了暴露一大堆本就不需要的参数信息等冗余内容，其实 Anthropic 推出 skills 一定程度上就是想弥补 MCP 的设计缺陷。

### 2.2 Reduce 减少

减少上下文的关键是压缩机制，好的压缩机制可以做到压缩大小的同时可以按需恢复，最大限度上减少 information loss。压缩机制可以分为 compaction 和 summarization。

Compaction 即指对上下文中的非必要信息进行“精简压缩”，如下图所示，当 LLM 写入一个文件时，文件的内容可能是比较多的，但是在多轮会话中，LLM 无需每次都知道刚刚自己写了什么，可能只需要知道是因为什么而写，写的内容的主题是什么就够了，如果再需要获取时，只需调用对应的工具读取文件内容即可。

![compact](/posts/context-engineering/compact.png)

Summarization 即指的是一旦对话超过某些轮数，再厉害的 compaction 也无济于事，只能对前文进行总结，总结后虽然会造成部分 information loss，但这是不可避免的。

### 2.3 Isolate 隔离

提高 Agent 能力的常见做法是将部分固定的、特定的的任务交给子 Agent 来做，从而一方面可以将一部分信息从上下文中去掉，另一方面可以防止上下文过多过杂导致的注意力混乱问题。当涉及到多 Agent 架构时，上下文的隔离是非常重要，尽管在多 Agent 架构中，有时可能会通过共享上下文来实现 A2A，但是这也要是以上下文不混杂为前提。

# 三、Manus 上下文工程

> 参考：https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus

Manus 是最早的 **通用 AI Agent** 的代表，作为通用型 Agent，其主要复杂度就是在上下文工程（最重要的当属工具设计），因此，经过不断踩坑与版本迭代，Manus 在 Context Engineering 上积累了许多经验。由于 Manus 是闭源的，我们只能在官方写的一些 blog 中学习到相关内容。

## 3.1 速度与性价比

作为生产环境的产品，速度和成本（性价比）是不可或缺的两部分，Manus 非常重视这部分，在文章中以 **Design Around** 来重点强调。基于 transformer 的特性，实现速度提升和成本降低的最直观也最重要的方法就是 **KV Cache**。

> If I had to choose just one metric, I'd argue that the KV-cache hit rate is the single most important metric for a production-stage AI agent.
>
>                                                                              ——— Context Engineering for AI Agents: Lessons from Building Manus

KV Cache 是 LLM 推理阶段的核心加速技术，本质是缓存历史 token 的注意力计算中间结果，避免重复运算。

**核心逻辑**：LLM 推理是逐 token 迭代的，每一步都需通过注意力机制计算【当前 token 与所有历史 token （即之前的 token）的关联】。KV Cache 会缓存之前所有 token 的「Key（键向量）」和「Value（值向量）」（注意力计算的核心中间产物），仅对新生成的 token 计算其「Query（查询向量）」，再直接与缓存的 KV 进行匹配，无需重新计算全部历史 token 的 KV。比如，如果连续解析两轮 token，而这两轮 token 的前 100 个 token 是一样的，只是后 10 个不同，那前 100 个就会命中缓存。

**为什么 KV Cache 很重要？**

上下文很大，而且随着每一轮迭代（多轮对话或工具调用）上下文会不断增长，这导致输入的 token 远远大于输出，Manus 在这方面的统计大约为 100:1。因此，处理输入是很重要的，解析如此庞大的上下文的速度是决定 AI 整体运行速度的最关键部分。而如果能命中缓存，速度和成本（如 claude sonnet，缓存 $0.3/M，未命中 $3/M）都会有所改观。

![KV Cache](/posts/context-engineering/kv-cache.png)

针对如何提高缓存命中率，Manus 给出了如下最佳实践：

- **保持整个上下文前缀稳定**：由于 KV Cache 的特性 —— 缓存的是当前 token 与所有历史 token （即之前的 token）的关联，如果某个 token 之前的 token 都不变，则该 token 就能命中缓存，所以最终给到 LLM 的输入一定要尽量保证前缀不变。

- **增量修改上下文**：保证之前的 action 或 observation 时不可变的，每轮只新增一部分上下文。这里需要工程层面的工作来保证，比如某些 JSON 序列化的库不保证 key 的顺序或者一些不稳定排序，都会导致缓存出错。

- **标记 Cache 断点**：某些模型和推理框架不支持自增前缀缓存，需要用户自己在上下文中插入断点。

## 3.2 工具设计

因为 Manus 是【通用 Agent】，所以其工具数量必然是非常多的，Agent 能力越强越通用，工具数量越多，这里面不仅包括 Manus 自己的工具，还有用户配置的自定义工具、MCP 等等。工具过多，Agent 就会变得愚蠢。

许多 Agent 的解决方案是参考 RAG 的思路，根据用户诉求和当前环境信息对所有工具进行检索，按需发送工具的信息给 LLM，Manus 之前也尝试过这种方法，但最终得出结论：除非万不得已，不要在多轮对话中动态地添加或移除工具。原因如下：

- 工具一般定义在整个给 LLM 输入的前面（这可能是 Agent 自己决定的，也可能是模型服务商决定的，比如 OpenAI API 的 tools），如果每轮工具都是动态的，KV 缓存会频繁失效；

- 输入给 LLM 的 Memory 中，存在之前的调用记录工具 A，但是工具 A 此次输入可能就没有了，这很可能会导致幻觉。

**解决方案**

有一个事实是避不开的，即工具数量确实多，而确实很多工具是本轮对话中 LLM 无需关注与调用的。Manus 通过使用可以感知上下文的壮态机来管理工具是否可用，通过覆写 token logits 来控制模型行为。这样既发送给 LLM 全量工具，又限制 LLM 可以调用哪些工具。

![mask](/posts/context-engineering/mask.png)

Token Logits 是模型生成每个词时的概率分数。覆写操作就是对不可用工具的对应的 token logits 进行修改，强制模型不选择这些 token：对于当前不可用的工具（如上图 Tool A），找到其对应的动作 token（比如调用 Tool A 的指令关键词），将这些 token 的 logits 设为极低值（或直接置 0）。这样，模型在生成动作时，几乎不会选择这些被掩蔽的 Token，只能从 “当前可用工具的 token” 中选择，从而避免 “schema 违规”（调用不存在的工具）。

能实现这一能力主要是因为大多数模型提供商和推理框架支持某种形式的**响应预填充**（让你通过自己填充模型的回复来限制模型的行为，举个极端例子，预先填充调用工具 A，模型就会调用工具 A），这就相当于是覆写 token logits。工具调用通常有三种类型，通过这三种类型即可实现工具的控制：

- **Auto**：无强制约束 Hermes 格式的预填充内容 `<|im_start|>assistant`

- **Required**：必须调用，但是工具无约束 Hermes 格式的预填充内容 `<|im_start|>assistant<tool_call>`

- **Specified**：指定工具子集，限制可用哪些，也就限制了不能用哪些 Hermes 格式的预填充内容 `<|im*start|>assistant<tool_call>{"name": "browser*`

## 3.3 上下文压缩

LLM 上下文窗口是有限的，而随着对话轮数的增多，上下文会越来越大，尤其是不可控的外部/非结构化数据（比如网页内容）比较容易超限。虽然像 Gemini 这种已经支持 1M token 的上下文，但是如果你的上下文真的有 999K，那它的回复质量将是很差几乎不可用的。还有就是输入越长，成本越高。

![compact-out](/posts/context-engineering/compact-out.png)

因此，要解决这个问题，上下文截断/压缩是不可避免的，正如 langchain 中的几种 Memory 的设计：

| memory 类型                        | 定义                                                                                           |
| ---------------------------------- | ---------------------------------------------------------------------------------------------- |
| buffer window memory               | 最近的 K 轮对话（截断）                                                                        |
| conversation summary memory        | 随着聊天不断生成和更新对聊天记录的摘要（压缩）                                                 |
| conversation summary buffer memory | 如果上下文历史过大时就切换到 summary，如果上下文比较小时就使用原始的聊天记录（截断与压缩混合） |

但是，过度压缩和不可逆压缩都是具有风险的，很有可能会导致信息丢失且不可恢复。

> From a logical standpoint, any irreversible compression carries risk.
>
>                                                                       ——— Context Engineering for AI Agents: Lessons from Building Manus

Manus 的解决方案是使用文件系统作为上下文的载体，文件系统的优点：

- 无限 size

- 自然持久化存储

- Agent 可以自己直接读取

![fs](/posts/context-engineering/fs.png)

上下文压缩不可避免，但是把原始信息存放在文件系统，意味着发送给 LLM 的 Context 里有部分是摘要/索引，如果 LLM 觉得它需要某部分的详细信息，他可以通过读文件按需恢复被压缩的部分，是可逆的。

## 3.4 其他

### 3.4.1 重复

Agent 的工作流程就是逐步进行的，比如 plan-and-excute 范式，会设定一个计划，然后不断的执行并 review 自己的计划执行并进行 replan，以防止自己跑偏。这是一种操控注意力的方式，本质上是使用 LLM 自己的能力来补齐 LLM 自己的短板（使用自然语言来控制注意力，而不需要对 LLM 本身进行架构优化等）。

### 3.4.2 暴露错误

是 LLM 就会犯错，犯错是正常的，有的 Agent 的设计是不暴露错误，而重新让 LLM 再执行一遍，这样的效果远不如先把错误抛给它，再让它自我调整并执行一遍。

### 3.4.3 Few-Shot 优化

LLM 模仿能力很强，如果上下文中的 examples 都是类似的，就会导致 LLM 可能会过度模仿，即使这样不是最优/正确的。Manus 的解决方案就是多样化，就像我们改论文降重一样。

// 寻找压缩点
function findCompressSplitPoint(
    contents: Content[],
    fraction: number,
): number {
    if (fraction <= 0 || fraction >= 1) {
        throw new Error('Fraction must be between 0 and 1');
    }

    const charCounts = contents.map((content) => JSON.stringify(content).length);
    const totalCharCount = charCounts.reduce((a, b) => a + b, 0);
    // 需要将最老的 70% 压缩
    const targetCharCount = totalCharCount \* fraction; // 1 - 0.3 = 0.7

    let lastSplitPoint = 0; // 0 is always valid (compress nothing)
    let cumulativeCharCount = 0;
    for (let i = 0; i < contents.length; i++) {
        const content = contents[i];
        // 用户消息 & parts 中不包含 functionResponse 的项
        if (
            content.role === 'user' &&
            !content.parts?.some((part) => !!part.functionResponse)
        ) {
        // 当前已经读取到超过 70%的字符数，返回当前索引作为压缩分割点
        if (cumulativeCharCount >= targetCharCount) {
            return i;
        }
        lastSplitPoint = i;
        }
        cumulativeCharCount += charCounts[i];
    }

    // We found no split points after targetCharCount.
    // Check if it's safe to compress everything.
    // 压缩全部
    const lastContent = contents[contents.length - 1];
    if (
        lastContent?.role === 'model' &&
        !lastContent?.parts?.some((part) => part.functionCall)
    ) {
        return contents.length;
    }

    // Can't compress everything so just compress at last splitpoint.
    return lastSplitPoint;
}

可以看到，寻找压缩点时，只会选择纯用户输入，本质是避免压缩操作切割关键对话逻辑，纯用户输入是对话的 “天然分段点”（用户每一次输入都是一个独立意图），以它为压缩点，能保证截断后的上下文完整保留 “用户意图 → 模型响应” 的完整链路。
